## Scope

v2.7.0은 **Phase 2C+ close-out tag**입니다. Pivot 2026-04-19가 예고한 세 가지 재측정 결과 시나리오 중, 5-pair preliminary 결과 기준으로 **시나리오 C ("변동 없음 → 2C+ 전략 재검토")**에 해당하는 것이 확인된 상태에서의 공식 릴리즈입니다. 이 tag는 "detection 보강 4-6주 작업이 끝나서 multi-agent로 넘어간다"는 식의 승리 서사가 아니라, **"2C+ 4항목을 실측했고, Gate 1/3은 2C+ 내부 작업으로 도달 불가임을 정직하게 기록한다"**는 close-out입니다.

## Gate-by-gate 진단 & 해결 경로 (Gate Diagnosis Matrix)

각 gate에 대해 (a) 무엇을 재는가 (b) 왜 FAIL인가의 정확한 원인 (c) 해결에 필요한 작업 (d) 어느 트랙에 배정되는가 — 4축 정리.

### Gate 1 — Detection Recall ≥ 0.40 [FAIL 0.143]

- **정의**: pair-eval corpus에서 vulnerable side가 target CVE를 잡고 patched side는 안 잡은 비율. `scripts/score_pair_corpus.py` 기반.
- **왜 FAIL**: SCOUT의 CVE detection이 SBOM → NVD CPE 매칭 기반인데, pair corpus의 target CVE 6개 중 5개가 vendor-specific CGI/HTTPD 바이너리의 메모리 손상 버그라 SBOM에 등장하는 component (openssl/dnsmasq/uclibc/libpthread) 매칭으로는 구조적으로 접근 불가. r7000의 CVE-2017-5521 (OpenSSL)만 잡힘.
- **해결 작업**: (i) taint-propagation 증거로부터 독립 rule_id로 vulnerability finding을 발행하는 `findings.py` 로직 신설, (ii) emulation-backed dynamic validation으로 reachability 확정, (iii) 코드레벨 interprocedural taint (현재 intra-function) 강화.
- **트랙 배정**: **external 트랙 (갈래 B detection engine 재설계)**. SCOUT Core 내에서는 Phase 2D'가 deferred 상태이므로 본 repo의 next-release scope 밖. OOB Local LLM AEG 트랙에서 자연 흡수 가능.

### Gate 2 — Evidence Tier 변별력 ≥ 2 nonzero TP tiers [FAIL 1]

- **정의**: fp_verification이 TP로 확정한 alerts가 서로 다른 `evidence_tier` (SYMBOL_COOCCURRENCE / STATIC_CODE_VERIFIED / STATIC_ONLY / PCODE_VERIFIED) 를 가진 개수.
- **왜 FAIL**: 2C.3 evidence_tier **필드** rollout은 완료됐지만, P-code taint engine의 실전 robustness가 낮아 PCODE_VERIFIED tier의 alert이 adversarial_triage 단계에서 거의 모두 downgrade. 실제 TP로 확정된 alert이 `symbol_only` tier에만 몰림.
- **해결 작업**: P-code taint를 truly interprocedural로 확장 (현재 intra-function 제약), adversarial_triage 프롬프트를 tier-aware 판정으로 개선, STATIC_INTERPROC tier에서 TP 생성 빈도 측정.
- **트랙 배정**: **external 트랙** (P-code engine 재설계는 갈래 B 작업). 부분적으로는 Phase 2D.2 multi-agent exploit chain에서 자연 흡수 가능 — 하지만 2D는 deferred.

### Gate 3 — Finding Diversity Index < 0.5 [FAIL 1.000]

- **정의**: `pair_eval_findings.csv`의 `finding_id` 컬럼에서 최다 ID 점유율. 14/14 전부 단일 ID면 1.000.
- **왜 FAIL (핵심)**: `findings.py`의 `choose_primary_finding()` 로직이 confidence 최고 finding 하나만 선택. 모든 firmware에 webserver + exec sink가 공존 → synthesis stage가 생성하는 `aiedge.findings.web.exec_sink_overlap` 합성 finding이 항상 최고 confidence → 14/14 전부 동일 ID. LARA sources (0→21-86 hits)나 LATTE slicing이 evidence를 풍부하게 해도 **선택되는 finding ID 자체는 변하지 않음**.
- **해결 작업**: (i) `findings.py`에 taint evidence 기반 독립 rule_id (`taint.cmd_injection` / `taint.buffer_overflow` / `taint.path_traversal`) 발행 로직 신설. 단 이것만으론 "cosmetic labeling splitting"으로 보일 위험 → 실제 detection quality 개선 증거와 엮여야 함. (ii) 대안: diversity metric 자체를 finding-ID 단위에서 evidence-level 단위 (source_method / sink_category / evidence_tier 조합) 로 재정의 — 1-2주 작업, 단 gate 재설계 vs. cosmetic dodge 경계 애매.
- **트랙 배정**: **Phase 3' 연구 트랙 (후속 detection 연구)**. Pivot doc의 Option C를 Phase 3'으로 재분류함에 따라, SCOUT Core의 v2.7.x scope가 아니라 갈래 A 트랙과 병렬 진행되는 별도 연구 작업.

### Gate 4 — Dedicated Reviewer Rerun ≥ 1/N success [PASS 14/14] ✅

- **정의**: 전용 reviewer lane의 pair set N runs 중 완료된 개수.
- **왜 PASS (의미 있는 승리)**: 2026-04-19 분석에서 "4 SCOUT analyze 동시 실행 + OOM 가설"로 `run_index rows=0` 상태 정체됐던 이전 lane 문제 완전 해결. 이번 세션에서 도입한 (a) `run_pair_eval.py` stdout 실시간 파일 redirect, (b) 순차 launcher + watcher handoff, (c) `nohup setsid` detach 조합으로 **12시간+ long-running job이 SSH 끊김에도 안정적 완주**.
- **해결 작업**: 이미 완료. 본 release의 scripts/ 아래 4개 launcher + watcher (`run_reviewer_sequential.sh`, `run_reviewer_codex_latte.sh`, `run_reviewer_claude_latte.sh`, `watch_codex_then_latte.sh`) 가 그 인프라.
- **트랙 배정**: **완료. 갈래 A SaaS 트랙 (3'.2)의 job queue 인프라 기반**. Phase 3'.2 CRA Audit SaaS의 multi-tenant async job 처리 설계에서 본 안정성이 직접 자산으로 활용됨.

### Gate 5 — Pair Corpus Size ≥ 10 [FAIL 7]

- **정의**: `benchmarks/pair-eval/pairs.json`에 등록된 (vulnerable, patched) pair 수.
- **왜 FAIL**: 단순 data shortage. d-link 4 + netgear 1 + tp-link 2 = 7 pair만 등록된 상태.
- **해결 작업**: **2C+.4 Vendor extraction chain 확장 (5종 — DIR-859 / RT-AC68U / WRT1900ACS / DIR-878 + 1종)**. 5 pair 추가 → corpus 7→12 → Gate 5 해제. 각 vendor 포맷별 extraction 로직 (`extraction.py`, `vendor_decrypt.py`) 확장 필요.
- **트랙 배정**: **v2.7.1 scope (next release)**. Phase 2C+의 유일한 pending 항목. Scenario C 확정으로 Gate 1/2/3은 deferred이지만 Gate 5는 compliance track의 corpus-credibility 관점에서도 의미 있어 여전히 유효한 작업.

### 종합 매트릭스

| Gate | 원인 카테고리 | 해결 난이도 | 트랙 | 예상 타임라인 |
|------|-------------|-----------|------|-------------|
| 1 Recall | 구조적 detection 한계 (SBOM-CPE matching) | High | External (갈래 B / OOB AEG) | 6-12개월+ (Phase 2D' deferred) |
| 2 Tier | P-code engine robustness | Medium-High | External | 별도 research track |

**이 매트릭스의 의미**: 5 gate 중 **이번 release(v2.7.0)에서 완결**은 Gate 4. **next release(v2.7.1)에서 완결** 목표는 Gate 5. **SCOUT Core scope 내 deferred**는 Gate 1/2/3. 이는 Pivot 2026-04-19의 Option D (Phase 2D' deferred, 갈래 A 올인) 결정을 gate-level에서 구체화한 것이며, 각 gate의 개별 해결 경로가 명시적으로 기록된 상태에서 release tag를 발행함.

## Release note 권장 문구 (draft)

> v2.7.0 is a close-out release for Phase 2C+ (detection reinforcement) as defined by the 2026-04-19 Direction Pivot. Four of the five 2C+ items (LATTE backward slicing, LARA pattern-based source identification, sink coverage expansion, finding diversity gate) are merged. A reviewer-lane measurement run under these changes shows that Phase 2D Entry Gates 1 and 3 (recall ≥ 0.40, finding diversity index < 0.5) cannot be cleared by the 2C+ workstream alone, because all current vulnerability findings are emitted as a single synthesis-stage finding ID regardless of how many source/sink evidence items the taint layer produces. The pivot roadmap's scenario-C (re-evaluate 2C+ strategy) is hereby entered. See `wiki/projects/scout-direction-pivot-2026-04.md` and `docs/v2.7.0_release_plan.md` for the full rationale.
