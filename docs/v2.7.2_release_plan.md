# v2.7.2 Release Plan — Phase 2C++ detection engine integrity patch

> Status: **shipped** (2026-04-24). Commit chain: `9488b8b` (Phase 2C++.1 Gap C)
> + `36ea517` (Phase 2C++.2 Gap B).

## Scope

v2.7.2 closes the two *low-cost, high-value* items from the v2.4.0 external
review (`docs/upgrade_plz.md`) that had been partially addressed in v2.4.1
but left cosmetic residues in the tree. It is **not** a behavioural pivot.
Pair-eval re-measurement is explicitly deferred because the expected
scorecard movement is ~0 (see "Why no re-measurement" below).

- **In**: Gap C (`DECOMPILED_COLOCATED_CAP = 0.45` promoted to a named
  constant, 5-tier cap ladder), Gap B (dead `trace_pcode_forward()` helper
  removed, unreachable `else: addr_diff` fallback removed, `source_api_name`
  made required), new pinning tests in
  `tests/test_ghidra_dead_code_removed.py`, docs updates in
  `docs/confidence_semantic_break_v2.6.md`.
- **Out**: Gap A (interprocedural taint, 2-3 weeks, handled in a later
  session that will also do the pair-eval re-measurement); DIR-878
  SHRS inner-bin decryption (separate session); any change to
  `findings.py`'s single-synthesis selection (external track per Pivot
  Option D).

## Why the v2.4.1 "fix" left residues

Git blame on the primary Gap B site (`ghidra_scripts/pcode_taint.py`) shows
two sibling commits 41 minutes apart on 2026-04-11:

| Commit | Time | Role |
|---|---|---|
| `6a8e340` | 16:13 | v2.4.0 — introduced the P-code taint engine with `addr_diff > 16` as the primary CALL-matching gate. This is the path the external review audited. |
| `3352783` | 16:54 | v2.4.1 — commit message reads `"fix: interprocedural taint, confidence calibration, **addr_diff removal**"`. Added `source_api_name` parameter and moved callee-name resolution to the primary path. However the offset gate was *demoted to an `else` branch*, not physically deleted. |

The `else` branch then became unreachable at runtime because `run()` always
calls `_trace_forward_pcode(..., source_api_name=source_api)` (see line
291-294, unchanged since 6a8e340). A parallel story played out in the
pyghidra fallback path: `ghidra_analysis.py` kept a `trace_pcode_forward()`
helper that the surrounding `_PYGHIDRA_SCRIPT` never invokes, because the
Strategy 1 inline loop around line 525-587 (also added in 6a8e340) took
over.

v2.7.2 physically removes both residues and pins the absence with
`tests/test_ghidra_dead_code_removed.py` so future grep-based review
doesn't re-report an already-fixed issue.

## Why no re-measurement

Pair-eval 12-pair `--no-llm` re-measurement would cost ~18 hours wall-clock
for ~0 expected scorecard movement:

1. **Gap B is already runtime-effective since v2.4.1.** This patch only
   removes the *code-shaped* residue; no production path changes.
2. **Gap C's new ceiling (`0.45`) only binds on `decompiled_colocated`
   traces.** That method label is emitted exclusively in one place:
   `ghidra_analysis.py:609`, inside `_PYGHIDRA_SCRIPT` Strategy 3. That
   path runs only when the pyghidra fallback is activated (analyzeHeadless
   failed for a binary and pyghidra was available). The current dev
   environment ships Ghidra 12.0.2 with analyzeHeadless on PATH and
   pyghidra installed; R7000 and the full 12-pair corpus historically run
   via analyzeHeadless, so the fallback rarely fires.
3. **Gate 1/2/3 FAIL is driven by `findings.py`'s single-synthesis
   selection bottleneck.** A 0.05 confidence shift on a rarely-emitted
   taint method cannot plausibly move any Gate 1/2/3 threshold because
   the synthesis finding (`aiedge.findings.web.exec_sink_overlap`)
   dominates both TP and FP sides, pinning recall/FPR and diversity.
4. **The Gap A decision gate (ROC movement evidence for Gap A investment)
   requires changes that *would* plausibly move scorecards** (cross-
   function taint that surfaces new finding IDs). v2.7.2 is not that
   kind of change.

If a later session implements Gap A and needs a 12-pair figure, the
re-measurement recipe is already in `docs/v2.7.1_release_plan.md` (same
commands, `benchmark-results/pair-eval-12pair-mixed/` baseline).

## Included changes

### Changed
- **Phase 2C++.1** — `DECOMPILED_COLOCATED_CAP = 0.45` promoted to a named
  constant in `confidence_caps.py`; `taint_propagation.py` imports and uses
  it instead of the inline literal `0.50`. Effective confidence drop for
  `decompiled_colocated` traces is `0.50 → 0.45` (-0.05). The 5-tier cap
  ladder is documented in `docs/confidence_semantic_break_v2.6.md`. Six
  new tests in `tests/test_confidence_caps.py` pin the value, the
  ordering, the neighbour-cap distances, and the `taint_propagation.py`
  import reference.

### Fixed
- **Phase 2C++.2** — legacy `addr_diff > 16` residues physically removed
  from `src/aiedge/ghidra_analysis.py` (dead `trace_pcode_forward()`
  helper inside `_PYGHIDRA_SCRIPT`) and from
  `src/aiedge/ghidra_scripts/pcode_taint.py` (unreachable `else` branch
  in `_trace_forward_pcode()`). `_trace_forward_pcode()`'s
  `source_api_name` parameter is now required (no default), formalising
  the invariant that has held since v2.4.1. Five new tests in
  `tests/test_ghidra_dead_code_removed.py` guard against reintroduction.

## Consumer migration

- If a downstream system pinned an ROC threshold at `0.50` for
  `decompiled_colocated` detections: retune to `0.45` to preserve
  pre-v2.7.1 recall on that specific evidence class. All other
  thresholds unaffected.
- No schema or artefact-format changes. `priority_score` weights
  unchanged. `cve_scan` uses `STATIC_CODE_VERIFIED_CAP=0.55`, also
  unchanged.
- No API changes to external modules; the `source_api_name` required-
  argument change is internal to `ghidra_scripts/pcode_taint.py` and
  its only in-script caller already passes the argument.
